# Python Runtime Quality Policy

Python 런타임 품질, 에러 처리, 로깅, 성능, 의존성, 보안 기준을 정의한다.

## 에러 처리

- 조용히 실패하지 않는다.
- 복구 가능한 오류는 명확한 예외나 결과 타입으로 표현한다.
- 외부 리소스 접근 실패는 원인을 보존해서 로깅한다.
- 너무 넓은 `except Exception`은 금지한다. 필요하면 즉시 재발생하거나 구체적 예외로 감싼다.
- 사용자에게 보여줄 메시지와 개발자용 로그 메시지를 분리한다.

## 로깅

- `print()`는 CLI 출력이나 임시 디버깅 외에는 사용하지 않는다.
- 애플리케이션 로그는 표준 `logging` 모듈을 사용한다.
- 로그에는 개인정보, 원본 영상 프레임, 민감한 경로를 남기지 않는다.
- 반복 프레임 처리 루프에서는 과도한 로그를 피한다.

권장 레벨:

- `DEBUG`: 개발 중 상세 상태, 프레임별 진단 정보
- `INFO`: 앱 시작/종료, 모델 로딩 완료, 주요 설정
- `WARNING`: 성능 저하, 카메라 일시 실패, fallback 사용
- `ERROR`: 모델 로드 실패, 설정 오류, 복구 불가능한 런타임 오류

## 성능 기준

- 프레임 처리 루프에서는 불필요한 객체 생성, 동기 I/O, 큰 배열 복사를 피한다.
- NumPy/OpenCV 연산을 우선 사용하되, 가독성을 크게 해치지 않는다.
- 성능 최적화는 측정 결과를 근거로 한다.
- 추론, 후처리, UI 렌더링 시간을 분리해서 측정할 수 있게 설계한다.
- 목표 성능은 하드웨어, 모델, 입력 해상도와 함께 기록한다.

## 의존성 정책

- 표준 라이브러리로 충분한 기능은 새 의존성을 추가하지 않는다.
- 새 의존성은 다음 기준을 만족해야 한다.
  - 유지보수가 활발한가
  - 라이선스가 프로젝트에 적합한가
  - 온디바이스 환경에서 설치와 실행 비용이 과하지 않은가
  - 대체 가능한 표준/기존 의존성이 없는가
- 개발 도구 의존성은 런타임 의존성과 분리한다.

## 보안과 개인정보

- 영상/이미지 데이터는 기본적으로 로컬에서만 처리한다.
- 원본 프레임이나 추론 결과를 저장할 때는 사용자 동의와 명확한 설정이 필요하다.
- API 키, 토큰, 로컬 절대 경로는 코드에 하드코딩하지 않는다.
- 설정은 환경 변수나 별도 설정 파일로 분리하고, 비밀값은 Git에 커밋하지 않는다.
